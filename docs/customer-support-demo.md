# Customer Support Demo

This is the first complete use case.

The goal is to ingest a full customer-support agent session and return a structured context package that a next support agent can use safely.

## Fixture

The demo fixture is:

```text
examples/customer-support-session.json
```

It contains:

- Support refund policy
- Customer profile
- Billing issue
- Tool lookup output
- Decision not to promise refund
- Active entitlement error
- Escalation state
- Next action
- Hidden expected facts for evaluation

## Run The API

```bash
DATABASE_URL=data/support-e2e.sqlite npm run start
```

## Run The End-To-End Demo

In another terminal:

```bash
npm run demo:support
```

The script:

1. Creates a session through the API
2. Ingests every event in the fixture
3. Calls `POST /v1/sessions/:sessionId/context-package`
4. Prints the resulting package summary

Expected output includes:

- `customer.name: Maya Chen`
- `issue.productArea: Billing`
- `issue.status: escalated`
- `escalation.targetTeam: Billing Ops`
- `operations` containing:
  - `keep_verbatim`
  - `externalize_for_retrieval`
  - `extract_active_error`

## Context Package Shape

The customer-support package includes:

```text
customer
issue
preservedInstructions
policyConstraints
troubleshooting
decisions
escalation
nextActions
runtimeContext
compactionPlan
externalReferences
metrics
```

The package is more useful than a plain summary because it separates operational support state from the compressed runtime context.

## Run Evaluation

The main cross-use-case evaluation is now:

```bash
npm run eval:accs
```

That compares support, coding, and voice fixtures against simple baselines plus `rolling_summary_recent`.

The support-specific legacy evaluator is still available:

```bash
npm run eval:support
```

This compares:

### Generic Support Summary

A generic first/last summary baseline.

### Adaptive Context Package

The actual context package generated by the API.

The evaluator checks whether expected support facts survived:

- Customer name
- Account ID
- Refund policy constraint
- Duplicate invoice IDs
- Active billing-page error state
- Next action

Expected result:

```text
adaptive_context_package wins
```

The current run scored:

```text
Adaptive recall: 6 / 6
Generic support summary recall: 5 / 6
```

The generic support summary lost the active entitlement state because it dropped `billing_portal_v2=false`.

## Web UI

The web UI includes this fixture as the **Support** demo. With the API and UI running, choose **Support**, click **Run package**, then click **Evaluate**.

The UI also includes **Coding** and **Voice** fixtures. Those paths use generic compaction rather than the customer-support package endpoint.
